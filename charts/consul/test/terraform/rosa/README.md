# ROSA Terraform

This module creates one or more isolated AWS VPC pairs, peers each pair, provisions the basic routing needed for cross-VPC connectivity inside each pair, and outputs the `rosa` commands needed to create the clusters manually into the generated BYOVPC subnet sets.

It is designed for Consul multi-cluster testing where the clusters need direct cross-VPC reachability for the Consul server plane.

## Prerequisites

1. `terraform`, `aws`, `rosa`, and `jq` must be installed.
2. `aws` must already be authenticated for the target account.
3. `rosa login` must already be completed.
4. The AWS account must be prepared for ROSA STS clusters. The generated commands use `rosa create cluster --sts --mode auto`.
5. The chosen VPC, service, and pod CIDRs must not overlap.
6. When using `cluster_pairs`, all pairs share the same `primary_region` and `secondary_region` provider settings.
6. When `az_count = 1`, the module builds a single-AZ ROSA layout and passes one private plus one public subnet per cluster. When `az_count > 1`, it adds `--multi-az` and passes all generated subnets for multi-AZ BYOVPC creation.

## What It Creates

1. For each configured pair: a primary AWS VPC with public and private subnets.
2. For each configured pair: a secondary AWS VPC with public and private subnets.
3. For each configured pair: an AWS VPC peering connection with routes in both directions.
4. Output values containing manual `rosa create cluster` commands and bootstrap scripts per pair.
5. Each generated command uses all subnets for the selected topology: one private plus one public subnet for single-AZ, or one private plus one public subnet per AZ for multi-AZ.
6. Output values containing a per-pair worker security group ingress command and bootstrap script that waits for readiness before applying it.

## Usage

```bash
cd charts/consul/test/terraform/rosa
terraform init
terraform apply \
  -var primary_cluster_name=consul-rosa-east \
  -var secondary_cluster_name=consul-rosa-west
terraform output -raw primary_rosa_create_command
terraform output -raw secondary_rosa_create_command
terraform output -raw rosa_cluster_bootstrap_script > rosa-bootstrap.sh
bash rosa-bootstrap.sh
```

For multiple isolated pairs in one apply, use `cluster_pairs` and the map outputs:

```bash
cd charts/consul/test/terraform/rosa
terraform init
terraform apply -var-file=two-pairs-418-419.tfvars.example
terraform output -json cluster_pair_details
terraform output -json rosa_create_commands
terraform output -json rosa_cluster_bootstrap_scripts
```

## Test Bed Example

For an east/west ROSA layout matching the current test-bed naming convention, copy the example tfvars and adjust CIDRs only if they overlap with existing routed networks:

```bash
cd charts/consul/test/terraform/rosa
cp test-bed-east-west.tfvars.example test-bed-east-west.tfvars
terraform init
terraform apply -var-file=test-bed-east-west.tfvars
terraform output -raw primary_rosa_create_command
terraform output -raw secondary_rosa_create_command
terraform output -raw rosa_cluster_bootstrap_script > rosa-bootstrap.sh
bash rosa-bootstrap.sh
```

The example file sets:

1. `primary_cluster_name = "test-bed-east"`
2. `secondary_cluster_name = "test-bed-west"`
3. `primary_region = "us-east-2"`
4. `secondary_region = "us-west-2"`
5. non-overlapping example VPC, pod, and service CIDRs
6. `az_count = 1` for a single-AZ test bed
7. `--channel-group stable` for both ROSA clusters

## Two Pair Example

If you want one isolated 4.18 pair and one isolated 4.19 pair, use [two-pairs-418-419.tfvars.example](two-pairs-418-419.tfvars.example).

That example creates two independent pairs, for a total of four clusters:

1. `rosa418`: OpenShift 4.18 east/west pair
2. `rosa419`: OpenShift 4.19 east/west pair

Each pair has its own:

1. Primary and secondary VPC CIDRs
2. Pod CIDRs
3. Service CIDRs
4. VPC peering connection

No routes or peering are created across different pairs, so each pair remains independent.

## Important Notes

1. This module discovers the ROSA node security groups from the cluster infra ID, VPC, and ROSA cluster tags rather than assuming a fixed `worker-sg` name.
2. Run `terraform output -raw worker_security_group_ingress_command` after both clusters become ready to open worker-node ingress from the peer VPC CIDR for Consul ports `8300`, `8301/tcp`, `8301/udp`, `8501`, and `8502`, or use `terraform output -raw rosa_cluster_bootstrap_script` to do create, wait, and ingress in one script.
3. When using `cluster_pairs`, use the map outputs `worker_security_group_ingress_commands`, `rosa_create_commands`, and `rosa_cluster_bootstrap_scripts`.
4. It does not manage ROSA cluster lifecycle or application-level Consul installation. It builds the network and emits the ROSA commands you run manually.
5. Before `terraform destroy`, delete the ROSA clusters manually so AWS resources can be removed cleanly.
6. If your environment requires private API endpoints, custom account roles, private link, or tighter load balancer security groups, pass additional flags through `primary_additional_rosa_args` and `secondary_additional_rosa_args`, or the per-pair overrides inside `cluster_pairs`.

## Suggested Overrides

Example with explicit extra ROSA flags:

```bash
terraform apply -var-file=two-pairs-418-419.tfvars
```
or 

```bash
terraform apply \
  -var primary_cluster_name=consul-rosa-east \
  -var secondary_cluster_name=consul-rosa-west \
  -var 'primary_additional_rosa_args=["--channel-group","stable"]' \
  -var 'secondary_additional_rosa_args=["--channel-group","stable"]'
```

## Delete cluster
```bash
rosa delete cluster -c test-bed-east-tf -y
rosa delete cluster -c test-bed-west-tf -y
```

## Terraform Destory (Once delete the cluster)
```bash
terraform destroy
```
